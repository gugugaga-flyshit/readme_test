# 绩效指标

绩效指标

---

## 一、 目标达成维度：OKR 信心指数 (60%)

在小团队中，OKR 是绩效的灵魂。不建议设定死板的完成率，而应评估对核心业务的贡献。

- 指标 1：OKR 平均信心指数
> 定义： 周期内（如单月/单季）核心 Key Results 的推进情况。
> 
> 公式： $FinalScore = \sum (KR\_Achievement \times Weight)$
> 
> 评估点： 是否完成了 MVP 版本的核心逻辑？是否支撑了前端三级联动等复杂组件的沉淀？
- 指标 2：关键里程碑达成率
> 评估点： 针对项目建设中定义的“发布窗口”，是否准时交付了具备上线质量的代码。

---

## 二、 工程质量维度：代码卓越度 (20%)

基于我们制定的 Code Review (CR) 和 后端 v2.1 规范，质量是不可逾越的底线。

- 指标 3：PR 质量与评审贡献
> 评估点：
> - 被拒绝率： 发起的 PR 是否因为违反规范（如没写 related_name、逻辑没放进 Service）而被多次打回？
> - 评审参与度： 作为评审人，是否能敏锐地指出他人的 SQL N+1 问题或组件封装缺陷？
- 指标 4：线上稳定性 (Production Stability)
> 评估点： 触发 hotfix/* 分支的频率。如果一个功能上线后频繁回滚或修复，说明单元测试或自测不达标。

---

## 三、 团队协作与沉淀：知识边际贡献 (10%)

这一维度是为了奖励那些“为了未来效率而工作”的人，直接对应我们的 Wiki 建设和组件沉淀规范。

- 指标 5：公共资产贡献值
> 评估点： * 本周期内沉淀了多少个 shared/ 目录下的公共组件？
> - 是否更新了核心业务逻辑的 Wiki 文档，减少了其他人的沟通成本？
- 指标 6：规范执行度
> 评估点： 提交记录是否符合 feat/fix 规范？Swagger 文档是否随接口同步更新？

---

## 四、 敏捷与响应维度：交付效率 (10%)

- 指标 7：PR 循环周期 (Cycle Time)
> 定义： 从代码第一次提交到合并进 develop 分支的平均时长。
> 意义： 反映团队协作是否存在阻塞（如评审不及时或任务拆分过大）。

---

## 五、 绩效评估建议表（3-10 人团队专用）

| 维度 | 指标示例 | 权重 | 采集工具 |
| :--- | :--- | :--- | :--- |
| **业务产出** | OKR / KR 达成情况 | 60% | 飞书 / Notion OKR 模板 |
| **代码质量** | 缺陷率、CR 反馈量、规范执行 | 20% | GitHub / GitLab PR 统计 |
| **技术沉淀** | 组件封装数量、Wiki 文档质量 | 10% | Git Repo / Wiki 历史 |
| **协作效率** | PR 处理时长、沟通主动性 | 10% | 团队互评 / 站会表现 |

---

## ⚠️ 避坑指南：给 Leader 的三条建议

- OKR 评分不直接等同于奖金： OKR 旨在鼓励挑战。如果拿 OKR 直接算钱，大家会倾向于把目标定得极低。奖金应基于 OKR 表现出的个人能力成长和业务影响力综合判定。
- 奖励“重构者”而非“埋头写 Bug 者”： 很多时候，一个清理了技术债、优化了 Django Manager 逻辑的人，比一个疯狂写出大量含 Bug 代码的人对团队更有贡献。
- 互评（Peer Review）的重要性： 10 人团队中，谁干得好，同事心里最清楚。建议加入 20% 的同伴互评权重。
